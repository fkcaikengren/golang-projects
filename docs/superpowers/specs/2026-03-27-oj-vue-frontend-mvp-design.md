# OJ Vue3 Frontend MVP 设计

## 背景

当前仓库已经具备 OJ MVP 后端的基础模型与 API 规划，目标是在同一仓库内新增一个独立前端工程，使用 Vite 脚手架搭建面向参赛用户的 OJ 站点壳子。

本次前端工作不追求完整联调，也不追求后台管理能力，而是优先完成可继续扩展的用户站点基础功能，覆盖：

- 首页入口与站点导航
- 登录 / 注册页面
- 题单列表页
- 题目列表页
- 题目详情页
- 提交记录页

本次明确使用以下技术栈：

- Vite
- Vue 3
- TypeScript
- Vue Router
- Pinia
- Naive UI
- TanStack Vue Query
- Axios

## 目标与非目标

### 目标

- 在仓库内创建独立前端目录，形成可运行的 Vue3 工程
- 建立面向 OJ 用户的站点式信息架构，而不是后台控制台风格
- 按后端 MVP API 结构抽象接口层，保证后续联调时不需要大改目录
- 使用 mock 数据或占位查询完成主要页面的浏览流程
- 保留最小登录态管理能力，为后续真实鉴权接入留出位置

### 非目标

- 不在本次实现真正的判题联调
- 不在本次实现完整代码编辑器能力，可先使用文本域占位
- 不在本次实现后台管理、题目维护、标签管理
- 不在本次实现完整权限模型、路由守卫细化、刷新 token 机制
- 不在本次实现 SSR、PWA、国际化

## 方案结论

采用“单仓独立前端应用”的方式，在仓库根目录下新增 `frontend/` 工程，作为 Vite 应用独立运行。前端采用 feature-first 目录组织方式，以业务能力拆分 `auth`、`problem-sets`、`problems`、`submissions` 等模块；服务端数据统一经由 TanStack Vue Query 访问，Pinia 仅保留用户登录态和少量全局 UI 状态。

相比只快速堆页面的方案，这个结构可以在保持 MVP 速度的前提下，避免后续接入真实接口时整体返工。相比现在就做完整生产联调能力的方案，这个方案更轻，符合“先完成基础功能”的目标。

## 信息架构

站点采用面向刷题用户的 OJ 结构，首要目标是让用户快速进入题单、题目与提交记录，而不是强调运营或后台信息。

顶层导航：

- 首页
- 题单
- 题库
- 提交记录
- 登录 / 用户入口

主要页面：

### 1. 首页 `Home`

用途：

- 提供平台入口与简短介绍
- 暴露热门题单入口
- 暴露最近可浏览题目入口
- 引导用户登录后进入做题流程

内容区块：

- Hero 区域
- 热门题单卡片
- 最近题目或推荐题目卡片
- “开始刷题”行动入口

### 2. 登录 / 注册页 `Auth`

用途：

- 提供基础账号进入能力
- 统一承载登录与注册表单

要求：

- 登录、注册可以分 Tab 或分路由，但视觉上保持为同一套身份页
- 成功后写入本地 token 与用户信息占位状态
- 初版允许接口未完全联调时使用 mock 成功响应

### 3. 题单列表页 `Problem Sets`

用途：

- 浏览题单卡片
- 通过题单进入对应题目列表

内容字段：

- 题单名称
- 描述
- 题目数量占位
- 进入按钮

### 4. 题库 / 题目列表页 `Problem List`

用途：

- 浏览题目列表
- 支持通过题单上下文进入

内容字段：

- 标题
- 难度
- 标签
- 用户状态占位，例如“未做 / 已尝试 / 已通过”

初版筛选能力：

- 题单上下文筛选
- 基础难度筛选占位
- 文本搜索占位

### 5. 题目详情页 `Problem Detail`

用途：

- 阅读题面
- 查看样例与标签
- 填写代码并发起提交

内容区块：

- 标题与难度
- 标签
- 题面描述
- 输入说明 / 输出说明
- 样例输入 / 样例输出
- 提示
- 代码输入区
- 提交按钮

说明：

- 初版代码输入区使用普通文本域或轻量封装组件，不引入复杂编辑器
- 提交成功后只要求展示提交动作结果和跳转或刷新记录，不要求完整实时判题体验

### 6. 提交记录页 `Submissions`

用途：

- 浏览个人提交记录

内容字段：

- 题目标题
- 语言
- 判题结果
- 提交时间

## 工程结构

建议目录如下：

```text
frontend/
  src/
    app/
    router/
    shared/
    features/
      auth/
      problem-sets/
      problems/
      submissions/
    widgets/
    pages/
```

约束如下：

- `app/`：应用启动、Provider 注册、全局样式、布局壳
- `router/`：路由定义与导航元信息
- `shared/`：通用组件、工具函数、API client、类型、mock 数据
- `features/`：按业务能力收拢 query、service、组件、局部 store
- `widgets/`：首页与列表页的复合展示区块
- `pages/`：路由级页面入口

该结构优先保证边界清晰，避免把页面、接口和状态全部塞进 `views` 或 `components` 中。

## 数据流设计

### 查询与缓存

- 所有服务端读请求统一通过 TanStack Vue Query
- Query key 按领域设计，例如 `['problemSets']`、`['problems', filters]`
- 页面不直接在组件里手写 Axios 请求

### 全局状态

Pinia 只承载少量全局状态：

- 当前用户信息
- token
- 是否已登录
- 可选的界面偏好

不将题单列表、题目详情、提交记录等服务端数据放进 Pinia，以免与 Vue Query 职责重复。

### API 层

按后端 MVP 能力拆分模块：

- `auth.ts`
- `problemSets.ts`
- `problems.ts`
- `submissions.ts`

API 函数签名尽量贴近真实后端返回结构，即使当前使用 mock，也不单独发明另一套前端协议。

## Mock 与联调策略

本次允许页面先依赖 mock 数据，但 mock 必须遵循真实 API 结构。

具体原则：

- 优先保留真实 API 调用函数
- 当接口不可用时，在 query 层或 mock adapter 层返回占位数据
- mock 数据集中管理，不散落在页面组件内部
- 页面组件不感知“真实接口”和“mock 接口”的切换细节

这样可以确保后续从 mock 切到真实接口时，主要改动集中在 API 层与配置层，而不是整页重写。

## 视觉与交互方向

整体视觉定位为“面向参赛用户的在线判题站点”，不是后台管理系统。

要求：

- 首页与题单页具备明确入口感
- 题目详情页强调阅读效率和信息层次
- 大量使用 Naive UI 基础组件，但外层布局和样式不能直接停留在默认组件堆叠
- 配色偏克制、理性、技术站点风格，避免后台控制台式高密度表格主导首屏

## 错误处理

初版错误处理要求保持简单但明确：

- 接口失败时给出统一消息提示
- 空数据页提供占位反馈
- 登录失败、提交失败等用户动作要有清晰提示
- 对未登录访问提交相关操作，可先给出轻量拦截或提示登录

## 测试与验证范围

本次前端基础功能应至少验证：

- 工程可正常安装与启动
- 路由跳转可用
- 首页、登录、题单、题目列表、题目详情、提交记录页面可访问
- Mock 数据下主流程可浏览
- 基础 lint / build 可通过

如果在本次实现中补充测试，优先覆盖：

- 认证状态最小 store
- 核心 API / query 封装
- 关键页面的基本渲染

## 实施边界

本设计只覆盖前端 MVP 骨架与基础用户流程，不包含后端接口增补设计。后续若进入真实联调阶段，应在当前 API 模块基础上逐个替换 mock 数据源，并补齐登录态校验、提交反馈与用户状态同步。

## 结论

本次前端应以 `frontend/` 独立 Vite 工程形式落地，优先完成 OJ 用户站点的页面骨架、业务目录结构、基础数据流和 mock 驱动流程。这样既能快速形成可运行界面，也能保证后续接入真实后端 API 时不需要推倒重来。
