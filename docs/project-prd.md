# 在线代码评测系统研发 PRD

## 1. 文档目标

本文档不是泛化的产品介绍，而是面向后续研发执行的总纲文档，用于统一以下内容：

- 项目中后期建设目标
- 前台与后台管理端的功能范围
- 判题系统与 Go 做题支持的研发边界
- 前后端协作约定
- 模块拆分、迭代顺序、验收口径

本文档用于指导后续设计、排期、任务拆分与开发实现。进入具体阶段后，可以从本文档继续拆出更细的专项设计文档。

## 2. 项目定位

项目目标是建设一个面向中文开发者和求职训练场景的在线代码评测系统，支持：

- 用户注册、登录、做题、查看历史提交
- 浏览题单、按标签和难度筛选题目
- 在线编写代码并提交判题
- 后台管理题目、题单、标签、测试用例、提交记录、用户状态
- 后续逐步支持真实判题能力，而不只是同步占位判题

系统中期的产品形态包括两个端：

- 前台用户端：面向做题用户
- 后台管理端：面向运营、内容维护、平台管理人员

## 3. 当前状态

截至当前版本，后端已具备 MVP 级基础能力：

- 用户邮箱注册与登录
- 题单列表与详情接口
- 题目列表与详情接口
- 提交记录写入与用户题目统计更新
- 基于同步占位逻辑的提交判题

当前仍未完成的关键能力：

- 前端页面与交互系统
- 后台管理端
- 真正的多语言判题执行链路
- Go 代码编译运行与测试用例判定
- 更完整的权限体系、审计、运营能力

## 8. 判题系统需求

### 8.1 判题系统目标

后续判题系统不再停留在“代码包含 return 即 accepted”的占位逻辑，而要具备真实执行能力。

判题系统中后期目标：

- 支持代码编译
- 支持执行测试用例
- 支持超时、内存超限、编译错误、运行错误、答案错误等状态
- 支持标准输出比对
- 支持隐藏测试用例
- 支持异步判题

### 8.2 判题流程

标准流程建议为：

1. 用户提交代码
2. 提交记录入库，状态为待判题
3. 提交任务进入判题队列
4. 判题服务拉取任务
5. 拉起对应语言执行环境
6. 编译代码
7. 逐个测试用例执行
8. 汇总结果并写回提交记录
9. 更新用户题目统计

### 8.3 判题状态

系统至少需要支持以下状态：

- pending
- judging
- accepted
- wrong_answer
- compile_error
- runtime_error
- time_limit_exceeded
- memory_limit_exceeded
- system_error

### 8.4 判题架构建议

后端架构建议拆为：

- API 服务：处理前端请求
- Judge Worker：执行判题任务
- Queue：承载异步任务
- Sandbox：隔离执行代码

在系统早期可以先不拆独立服务，但代码结构上要为未来拆分保留边界。

### 8.5 判题模块拆分建议

为了让后续研发能并行推进，判题系统建议拆为以下模块：

- `submission intake`：接收提交并入库
- `judge dispatcher`：派发待判题任务
- `judge executor`：执行编译与运行
- `result aggregator`：汇总单次提交结果
- `stats updater`：更新用户题目统计
- `judge audit`：记录日志与错误明细

各模块要求：

- 输入输出边界清晰
- 可以单独测试
- 可以逐步替换实现

### 8.6 判题结果数据要求

真实判题后，单次提交至少需要能追踪以下信息：

- 提交源码
- 语言
- 编译输出
- 最终状态
- 总耗时
- 峰值内存
- 测试用例通过数
- 测试用例总数
- 系统错误信息

如果这些数据不能稳定保存，后续后台查看和问题排查会非常困难。

## 9. Go 做题支持需求

### 9.1 为什么优先支持 Go

项目本身后端技术栈为 Go，优先支持 Go 做题有以下价值：

- 平台自身技术栈一致，开发和调试成本较低
- 可以更早验证真实判题链路
- 更适合目标用户中的后端开发者群体

### 9.2 Go 判题支持范围

首版 Go 判题需要支持：

- 用户选择 Go 语言提交
- 生成标准源码文件
- 执行 Go 编译
- 执行生成后的程序
- 将标准输入传入程序
- 比对标准输出
- 返回编译日志、运行状态、耗时、内存等结果

### 9.3 Go 判题约束

需要明确：

- 支持的 Go 版本
- 单题时间限制
- 单题内存限制
- 是否允许第三方包
- 是否禁用网络访问
- 是否限制文件系统访问

推荐第一阶段约束：

- 固定 Go 版本
- 禁止网络访问
- 禁止外部依赖下载
- 仅允许标准库
- 受控临时目录执行

### 9.4 Go 题目适配方式

Go 做题的题目模式建议统一为 ACM 输入输出模式，原因是：

- 判题稳定
- 跨语言兼容性强
- 不依赖平台注入函数签名

后续如果需要类似 LeetCode 的函数签名模式，可以在题目层新增题目模板与包装器机制。

### 9.5 Go 判题阶段拆分

建议分 3 步建设 Go 做题支持：

1. `Go 占位升级版`
通过真实 `go build` 和简单执行链路，先打通最小闭环。

2. `Go 测试用例版`
引入标准输入输出测试用例、结果比对、编译错误与运行错误区分。

3. `Go 沙箱版`
加入资源限制、超时中断、隔离执行、日志保留。

这样做的好处是每一步都能上线验证，而不是一次性做完整复杂判题系统。

## 10. 数据与领域模型要求

当前已落地的核心领域对象包括：

- users
- problem_sets
- problems
- tags
- problem_tags
- problem_set_problems
- submissions
- user_problem_stats

后续需要补充的核心表建议包括：

- problem_test_cases
- problem_languages
- judge_jobs
- judge_logs
- admin_users
- operation_logs

建议新增字段或附属实体时遵循以下规则：

- 面向前台展示的数据与面向内部执行的数据尽量分离
- 判题任务流转状态不要只靠单表字段堆叠
- 支持多语言的配置不要硬编码在题目正文中
- 题面、测试用例、语言配置之间要保持可独立演进

### 10.1 中后期建议补充的数据对象

为支持后续研发拆分，建议明确以下对象：

- `problem_test_cases`
  记录输入、输出、是否示例、排序、是否启用
- `problem_languages`
  记录题目支持语言、默认模板、编译命令、执行命令、资源限制
- `judge_jobs`
  记录异步判题任务状态、重试次数、执行节点、开始结束时间
- `judge_logs`
  记录编译日志、运行日志、系统异常信息
- `admin_users`
  记录后台用户与角色
- `operation_logs`
  记录后台关键操作审计信息

### 10.2 关键关系约束

后续研发中要保持这些关系不被破坏：

- 一个题目可以对应多个测试用例
- 一个题目可以支持多种语言
- 一个提交只能关联一个用户、一个题目、一种语言
- 一个判题任务只能服务于一次提交
- 一个普通用户与一个后台管理员身份默认不共享表结构

建议遵循原则：

- 用户可见数据与内部执行数据分表
- 高查询频率统计数据可冗余保存
- 所有判题任务必须可追踪
- 所有后台关键操作尽量保留审计日志

## 11. 前后端协作约定

### 11.1 前端技术栈

建议沿用当前约定：

- Vue 3.5
- Naive UI
- Pinia
- @tanstack/vue-query

### 11.2 API 约定

当前与后续接口应保持统一返回结构：

- `code`
- `message`
- `data`
- `request_id`

后续接口约定需要继续统一：

- 分页参数格式
- 列表筛选参数格式
- 错误码体系
- 登录态传递方式
- 管理端接口鉴权方式

建议具体约定如下：

- 列表接口统一使用 `page`、`page_size`
- 筛选参数统一使用 query string
- 创建接口统一返回新对象或关键标识
- 更新接口统一返回最新对象
- 删除接口统一返回空成功结构
- 管理端接口统一使用 `/api/admin/v1/*`
- 前台接口统一使用 `/api/v1/*`

### 11.3 接口分层建议

为了避免后续接口混乱，建议尽早按资源维度划分：

- 认证接口
- 题单接口
- 题目接口
- 提交接口
- 用户进度接口
- 管理端题目接口
- 管理端题单接口
- 管理端测试用例接口
- 管理端判题配置接口
- 管理端用户接口

### 11.3 权限要求

后续至少分为：

- 普通用户权限
- 管理员权限

后台管理接口必须与前台用户接口隔离，避免后续权限模型混乱。

## 12. 非功能要求

### 12.1 安全性

必须逐步满足：

- 密码哈希存储
- 登录鉴权
- 管理端权限控制
- 判题执行环境隔离
- 禁止用户代码直接访问宿主机敏感资源

### 12.2 可维护性

要求：

- 保持 `model -> repository -> service -> handler -> router` 分层
- 判题逻辑和 API 逻辑解耦
- 后续异步任务逻辑独立封装
- 前后台接口边界清晰

### 12.3 可观测性

后续需要逐步补充：

- 结构化日志
- 提交链路日志
- 判题耗时监控
- 队列积压监控
- 错误告警

### 12.4 性能要求

中后期需要明确的最低性能目标：

- 题库列表页接口在正常数据量下可稳定分页
- 题目详情页接口响应时间应适合页面首屏使用
- 提交操作应快速返回“已接收”或“已判题”结果
- 判题任务应支持排队与削峰，避免 API 请求被长耗时阻塞

### 12.5 测试要求

后续每个阶段都应至少覆盖：

- 核心 service 单元测试
- 关键路由或 handler 测试
- 关键流程集成测试
- 判题链路专项测试

没有测试的模块不应视为已稳定交付。

## 14.1 推荐的下一步开发拆分

如果按最小阻力推进，建议后续研发直接按以下顺序开工：

1. 前台 Web 用户端
2. 后台管理端基础版
3. 测试用例与判题配置数据层
4. Go 真实判题链路
5. 异步判题与可观测性

原因：

- 前台和后台先落地，能尽快验证内容运营与用户体验闭环
- 判题系统放在内容管理能力之后，会减少返工
- Go 真实判题适合作为第一门语言试点

## 14. 风险与待决策项

当前已知关键风险：

- 真实判题涉及执行隔离与宿主机安全
- Go 判题的编译与执行资源控制需要严谨设计
- 后台管理与前台权限边界如果不提前设计，后续容易返工
- 题目内容格式若设计过于随意，前台渲染和后台编辑会长期混乱

待决策项：

- 管理端是否复用前台项目还是独立项目
- 判题队列采用哪种实现
- Go 沙箱方案采用进程隔离、容器隔离还是外部执行节点
- 测试用例导入格式采用文本还是结构化 JSON
- 题面编辑采用 Markdown 还是富文本模型

## 15. 验收标准

后续各阶段设计与开发必须至少满足以下验收原则：

- 每个阶段有明确的接口或页面交付物
- 每个核心流程都有可执行验证路径
- 前台与后台都具备最小可用闭环
- 判题结果可追踪、可定位问题
- Go 做题能力不是“伪支持”，而是具备真实编译运行能力

### 15.1 前台验收标准

- 用户可完成注册、登录、浏览题单、浏览题目、提交代码、查看提交结果
- 页面具备基本可用性，不存在主流程断裂
- 主要页面在桌面端具备稳定布局
- 错误态、空态、加载态明确

### 15.2 后台验收标准

- 管理员可完成题目、题单、标签、测试用例的维护
- 管理员可查看提交记录和用户状态
- 管理端与前台接口权限隔离清晰
- 关键变更操作具备可追踪性

### 15.3 判题验收标准

- Go 提交能完成真实编译与运行
- 系统能区分编译错误、运行错误、答案错误、通过
- 每次提交有可追踪的日志或错误明细
- 用户题目统计与提交结果一致

## 16. 后续文档拆分建议

当进入具体开发阶段后，建议继续拆出以下专项文档：

- 前台页面与交互设计文档
- 后台管理端信息架构与页面设计文档
- 判题系统架构设计文档
- Go 判题执行链路设计文档
